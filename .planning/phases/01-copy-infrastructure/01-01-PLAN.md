<objective>
Build the typed Vietnamese copy foundation: a single `AppCopy` module that is the source of truth for user-facing strings, a derived `CopyKey` union that makes a bad key a build error, interpolated strings modeled as named-arg functions, a review-only glossary enforcing one term per concept, and a documented ripgrep migration recipe — all under `src/Common/Copy/`, importable as `@common/Copy`.

Purpose: Every later screen (wizard, empty-states, app-wide copy pass in Phase 5) is written through this module instead of hand-editing ~408 inline strings twice. This phase locks the structure; exact Vietnamese phrasing is validated with a household user in Phase 5.

Output: Three new files — `AppCopy.ts`, `Glossary.ts`, `index.ts` — plus an empirically demonstrated build gate (a wrong key / dropped arg fails `tsc --noEmit`, then reverted).
</objective>

<threat_model>
## Trust Boundaries

This phase adds a compile-time-only typed-constants module. There is no runtime logic, no user input, no network call, no auth, no data store, and no rendering introduced. The only meaningful boundary is authored-content → compiled bundle (copy strings become part of the shipped JS), which is not an attacker-controlled boundary.

| Boundary | Description |
|----------|-------------|
| author → source module | Developer authors static Vietnamese copy; no untrusted input crosses here. |
| source module → compiled bundle | `tsc`/craco compiles authored strings into the app; content is static and developer-reviewed. |

## STRIDE Threat Register

The attack surface is effectively nil for a static strings module. The one genuine content risk is documented honestly below; the remaining STRIDE categories do not apply and are recorded as accepted with rationale rather than inventing threats.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | Authored copy strings in `AppCopy.ts` / `Glossary.ts` / `emptyStates`/`error` messages | mitigate | Never embed the GitHub PAT or admin PIN (or any secret) in any copy or error string; reference credentials by presence/absence only. The Glossary review must flag any concept that references credentials. (PITFALLS.md "Security Mistakes".) |
| T-01-02 | Spoofing | n/a | accept | No identity, auth, or session code in this phase. |
| T-01-03 | Tampering | n/a | accept | No data store, no mutable runtime state; module is static `as const` content compiled into the bundle. No package installs (zero new dependencies → no supply-chain checkpoint needed; RESEARCH "Package Legitimacy Audit: Not applicable"). |
| T-01-04 | Repudiation | n/a | accept | No actions/logging introduced. |
| T-01-05 | Denial of Service | n/a | accept | Pure compile-time constants; no runtime execution path to exhaust. |
| T-01-06 | Elevation of Privilege | n/a | accept | No access-control or privileged operation introduced. |

No package-manager installs occur in this phase, so the supply-chain legitimacy gate and `T-{phase}-SC` checkpoint do not apply (verified: RESEARCH.md "Package Legitimacy Audit — Not applicable").
